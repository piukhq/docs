1.0 Transmission Protocol Overview

This document outlines the protocols by which remote client systems can connect to and communicate with the Givex host. The Givex host runs different services that will allow authorized client systems to connect to it. The different protocols facilitate integration for different client frameworks.

1.1 Dataconnect

This protocol defines requests and responses in a specific form. The form is a packet defined in the Dataconnect Packet Definition section below.

The protocol is based on delimited packets with variable length fields separated by defined field separators. It uses the ASCII character set which is reproduced as an appendix to this document for convenience. The packets are communicated over standard TCP/IP socket connections using TLS. This protocol is the most efficient of the protocols documented in this specification.

1.2 JSON-RPC

JSON-RPC uses lightweight name-value pairs to encode messages over HTTPS. Client libraries and utilities exist for many different platforms to enable rapid development. JSON-RPC is recommended particularly for environments where the size of data transmission can be an issue, such as mobile apps.

This protocol is described in detail at http://jsonrpc.org/

The JSON format is described in detail at http://www.json.org

Note that Givex uses version 2.0 of the JSON-RPC protocol.

1.3 XML-RPC

XML-RPC uses XML encoded messages over HTTPS. Client libraries and utilities exist for many different frameworks to enable rapid development.

This protocol is described in detail at http://www.xmlrpc.com

2.0 Transaction Flow

A transaction consists of a request sent from the client to the Givex host, and a response sent from the Givex host to the client.

2.1 Dataconnect

A request/response transaction consists of a request sent to the Givex host followed by a response sent back to the client.

This protocol is designed to operate on top of TCP/IP which is considered to be reliable. As such there are no retries or ACKs except as supplied by the TCP/IP protocol. This is the preferred protocol since end-to-end security is provided by the Transport Layer Secruity. The Givex host uses a public signed certificate. We recommend the use of TLS1.2 although we also support TLS1.1.

If no traffic is received by the host within 30 seconds then the host will disconnect.

Here is the flow of a basic request/response transaction.

CLIENT                                                HOST
------                                                ----
 |                                                     |
 |                   REQUEST PACKET                    |
 |  ------------------------------------------------>  |
 |                                                     |
 |                   RESPONSE PACKET                   |
 |  <------------------------------------------------  |
 |                                                     |

2.1.1 Dataconnect Dial-up (Dcdial)

If either dial-up or TCP/IP with dial-up backup will be supported then integration using the dcdial protocol is necessary. Dcdial operates over both dial and TCP/IP and includes ACK/NAK checking, LRC checksum verification and retry logic. This allows implementation of a single protocol to simplify switching between dial and IP connections.

A request/response transaction consists of a request sent to the Givex host followed by a response sent back to the client. After receiving the request/response packet, the host/client should send an ACK using the serial byte from the packet. The client should follow the ACK with two CAN characters if no further communication is to occur on the socket.

If two CAN characters in a row are received by either side of the connection, this signifies that the connection should be closed immediately. This is a signal that one side of the connection is having problems (such as too many timeouts as described below in the Packet Definition section) or has completed the transaction. It will close down the channel and will not accept further transmissions.

After each packet is sent, the sending system should wait 5 seconds for an ACK, as defined below. If the expected ACK is not received within 5 seconds, the sender should retry. If no response is received after 3 tries the sender should send two CANs and disconnect.

The side waiting for a packet should wait for up to 15 seconds. After 15 seconds of no activity it should send two CANs and disconnect.

CLIENT                                                HOST
------                                                ----
  |                                                     |
  |                     ENQ                             |
  |  <------------------------------------------------  |
  |                                                     |
  |             REQUEST PACKET (Serial byte: 5)         |
  |  ------------------------------------------------>  |
  |                                                     |
  |             REQUEST ACK (Serial byte: 5)            |
  |  <------------------------------------------------  |
  |                                                     |
  |             RESPONSE PACKET (Serial byte: 1)        |
  |  <------------------------------------------------  |
  |                                                     |
  |             RESPONSE ACK (Serial byte: 1)           |
  |  ------------------------------------------------>  |
  |                                                     |
  |                   CAN CAN                           |
  |  ------------------------------------------------>  |
  |                                                     |

3.0 Packet Definition

3.1 Dataconnect

The first character of the packet is STX to signal the start of the data.

Next is the actual data sent as a series of fields separated by FS. There is no limit on field sizes. The first field is a 3 digit number to specify the service. Older integrations may use the character representation of the service (case is irrelevant). The specific services and fields are described in the Services section below. All data is encoded using the ISO Latin 1 character set.

If the service request is preceded by a dot (.) then the transaction is assumed to be the last one and the host will close the TCP/IP connection after sending its response. This does not apply to printing functions which could return multiple responses. This facility is designed to accommodate 3201/IP gateways that may not be able to control the TCP/IP channel directly.

After the last field there is an ETX to signal the end of the data.

Everything that is not between the STX and ETX is considered comments and is ignored by the Givex host.

               +-------------+
               |  repeating  |
+-----+--------+----+--------+-----+
| STX | <data> | FS | <data> | ETX |
+-----+--------+----+--------+-----+

If the request packet is improper, i.e., the service is unrecognized, the host will respond with a message containing the text "What?" and disconnect.

If the request is a valid request/response type of packet, a return packet is sent with the transaction results. The transaction type will be echoed back to the client in the first field of the response.

The services and fields are defined in the next section.

Some fields use colon ":" or pipe "|" separators if they contain multiple child data.

Some fields can contain lists of data, and those fields in turn can contain sub-lists of data. The order of field separator precedence for multiple levels of lists is:

Dataconnect
=====================
FS   Field Separator
GS   Group Separator
US   Unit Separator
RS   Record Separator
VT   Vertical Tab
SO   Shift Out
SI   Shift In

For example, consider the compound field ecertOrderInfo. This field is actually a list of compound elements, with the different list elements being separated by GS, and the elements of the compound element being separated by US. The sub-element orderDetailInfo is another list of compound elements, each of which is separated by RS. The elements of orderDetailInfo are separated by VT. entryGreetingMessage is another list of compound elements, each of which is separated by SO. The elements of each entryGreetingMessage are separated by SI. This field could be represented like this

ref<US>name<US>email<US>date<US>
 ref2<VT>ref3<VT>qty<VT>denom<VT>to<SI>from<SI>body

or, with multiple orderDetailInfo and entryGreetingMessage elements, like this:

ref<US>name<US>email<US>date<US>
 ref2<VT>ref3<VT>qty<VT>denom<VT>to<SI>from<SI>body<RS>
 ref4<VT>ref5<VT>qty2<VT>denom2<VT>to2<SI>from2<SI>body2<SO>
                         denom2<VT>to2<SI>from2<SI>body2

or there could be multiple instances of that structure, separated by GS.

3.1.1 Dataconnect Dial-up (Dcdial)

The first character of the packet is SOH and signals the start of a packet. All data before the SOH and after the end of the packet (as described below) is discarded. This allows the client to use any end of line convention or even none at all.

The next character is a serial byte. The value used should be different for each packet sent and should be from the set of ASCII digits. Each side keeps its own sequence of values. This value should be unique within any 3 contiguous packets.

The next byte is STX to signal the start of the data.

Next is the actual data payload. This is the same data as described in The Dataconnect section above.

After the last field there is an ETX to signal the end of the data.

Following this is an LRC byte. Here is an example C function to calculate an LRC.

/* This function takes a string and calculates an LRC value / / It is assumed that the STX is already stripped */ int lrc(const char *str) { int lrc = 0;

while (*str)
    lrc ^= *str++;
    return lrc;
}

A properly formed packet will look like this:

                              +-------------+
                              |  repeating  |
+-----+--------+-----+--------+----+--------+-----+-----+
| SOH | Serial | STX | <data> | FS | <data> | ETX | LRC |
+-----+--------+-----+--------+----+--------+-----+-----+
                     |   LRC calculated on this   |
                     +----------------------------+

If the packet is improperly formed or the LRC check fails, the system will return a NAK otherwise an ACK followed by the serial byte in the received packet is returned. If neither is received within 5 seconds then it should be retried. If a packet is sent 3 times without receiving an ACK then the transmission channel should be closed.

If the packet is OK and it is a request/response type of packet, then after sending the ACK it is processed and a return packet is sent with the results. This packet is formed exactly like the original packet with the same request byte returned (always in upper case) except that the contents are composed of fields as defined in the "Services" section below and the serial byte is replaced with one generated by the Givex host.

3.2 JSON-RPC

To use JSON-RPC to communicate to Givex you instantiate a JSON-RPC object then call one of our services defined in section 5.

All our functions return a list of values, the length and contents of the returned list varies. The returned list may change in the future, however the new returns will always be appended to the end of the list, and existing returns will not be removed.

Givex JSON-RPC is based on our Dataconnect protocol. The functions in our API take the same parameters as the equivalent dataconnect request packets, and the returned lists are the same as the return packet, with the exception that the numeric service code is not included because this is not necessary, since it is defined in the method name being called.

The JSON-RPC service names are based on the Dataconnect service number with "dc_" prepended to it. For example "dc_906" is the service name for the Dataconnect Activate transaction.

Note
 Fields that contain lists of data with sub-lists should replace the Dataconnect field separators with programming language specific nested lists:

===========================================================

US   Unit Separator          Sublist of elements
RS   Record Separator        Sub-sublist of elements

ASCII control characters: 19999<US>19.99<US>3<GS>20000<US>4.99<GS>19999<US>19.99

4.0 Field Definitions

This section defines the dataconnect fields. Each service definition consists of a description and a list of the fields that make up the request and response packets. Any additional information related to a field’s usage by a particular service is noted and the potential error messages are provided.

In general, date-valued fields should be sent using the format defined in IETF RFC 3339. Date fields should be sent as a "fulldate" and timestamps should be sent as a "date-time" as defined in the RFC, except using a space rather than a T as the seprator. Timestamps which do not include a "time-offset" are normally interpreted as being in the time zone of the transacting location.

Boolean-valued fields should be sent using "t" for true or "f" for false. Historically, Givex has also accepted "true", "yes", any abbreviation of those two, "on", or "1" for true.

Language code:

This is a two-letter code in the ISO 639 international standard identifying the language of the error text in the response packet. The codes correspond to the following languages:

en = English
fr = French
es = Spanish
es-mx = Mexican Spanish
zh = Chinese
pt = Portuguese
sv = Swedish
de = German
it = Italian
tc = Traditional
ja = Japanese
da = Danish
nl = Dutch
tr = Turkish

For example, to receive error messages in French this field would be fr. Currently these are the only languages supported. Any other code in this field will default the language to English. Note: To maintain backwards compatibility with versions of this protocol previous to 2.5.0, the use of 0 and 1 to indicate English and French respectively will continue to be supported.

Transaction code:

This is a value specified by the client system. It is tracked by the host for auditing purposes and for validation of reversal. It is always echoed back to the client system as the second field (after the service code). No processing is done on this field by the Givex host.

User ID:

This is the user ID as provided by Givex. This field should be able to accommodate a 10 digit ID. If you want to include your operator ID for reporting purposes you can prepend it and separate the two with a colon. For example, if operator 4 was on terminal user 123 and you wished to track that you would put "4:123" as the user ID.

Password:

This is the alpha-numeric password assigned to the user, and is required for every service. This is usually programmed into the POS system. If you want to send the operator password as well, prepend it and separate it from the user password with a colon similar to the way that operator and user IDs are handled above.

Security Code:

This is an optional field when specified in a request. If omitted or included as a blank field it is ignored by the Givex host. If the code is included then the system will check it against the Security Code assigned to the card. This is typically used by systems that want to add an extra layer of security for online transaction processing. It is returned appended to the authorization code by certain services depending on merchant settings. The Security Code can be printed on receipts for customer reference, but it should not be stored by the POS.

Operator ID:

Although we can accept the operator ID as part of the User ID, we sometimes need to enter an operator ID separately. This happens when the operator entering their password has permission to perform operations related to other operators. While this is generally a numeric value, a function may allow the string "ALL" to refer to all operators in the store that the terminal is registered for.

Givex number:

This is the primary reference number for Givex certificates. This number can be printed on gift cards, stickers or transmitted to recipients by email. In the case of cards, it can also be encoded in a magnetic strip and printed as a bar code. This number can be alphanumeric and can also be mapped to clients existing imported ISO, ticket barcode or SUN numbers. This field can also be used to perform a phone number lookup. The first character in the field must be a # or + to indicate a phone number. The format of the phone number must conform to the ITU-T E.123 standard.

ISO-serial:

This is the ISO plus serial number of a Givex certificate. It consists of the card ISO and serial number separated by a dash, ex. 123456-12345

Voucher Number:

This is the primary reference number for a Loyalty Voucher that contains stored-value. This number can be printed on coupons, gift cards, stickers or transmitted to recipients by email. In the case of cards, it can also be encoded in a magnetic strip or printed as a bar code. In order for the Voucher Number to be redeemed, the Givex Number to which it is related must also be presented in the same transaction for validation.

Program-Specific Data (PSD):

We have sometimes also called this "AdditionalData". The PSD field is used to pass data which is specific to an integration. The format of the data is generally negotiated with Givex and requires custom development both in the Givex service and the integration. In some cases, more generally useful extensions are documented below.

PSD may be used to affect processing, or to request additional data in a response.

Amount:

Specify in the currency of the transaction location with decimal, eg. "123.00"

Units:

This value is entered by the POS system and is tracked by the Givex system for the points program. This could be the total number of items or some calculated unit value or anything else defined by the user. This can be used later when calculating the value to add to the card.

Bonus Points:

This value is entered by the POS system and is tracked by the Givex system for the points program. This could be Bonus points allotted to customer or some calculated unit value or anything else defined by the user. This can be used later when calculating the value to add to the card.

Givex transaction reference:

This value consists of an Givex Auth Code and an optional Security Code separated by a colon. For example, if the authorization code 12345 is obtained after a transaction on a card with a Security Code 09876,then "12345:09876" would be returned in this field.

Result:

This is a return code. If the service succeeded this will be zero otherwise it will be a code with meanings as described in the descriptions of the individual services.

Report type:

This is used to define the type of report requested. Currently this can be "1" for certificate or "2" for points.

Certificate balance:

Balance available on the certificate. This takes the current transaction into account if applicable. If the card is a CLP one and CWS customer has Allow Balance Sharing flag turned on - then remaining shared balance on other customer CLP card(s) will be included: 909 and 994 are affected with the logic

Certificate expiration date:

The date that the certificate is scheduled to or did expire. If the certificate has no expiry date then "None" is returned in this field. It will be formatted as YYYY-MM-DD.

Points balance:

The amount of points accumulated on this card.

Unit balance:

The amount of units accumulated on this card.

Loyalty Balance:

The portion of the certificate balance which is the result of points conversions. This field may not be returned depending on the merchant set-up.

Balance as of Effective Date or PSD:

If the Effective Date field contains a date in yyyy-mm-dd format, this field contains the balance the coupon will acquire on that date. If the effective date is already past, this field will be 0. A coupon with value 10 which becomes effective at some time in the future will show 0 in the balance field and 10 in this field. Once the effective date is reached, the coupon will show 10 in the balance field and 0 in this field, and after it’s been used, it will show 0 in both fields.

If the effective date field contains data which is not a date in yyyy-mm-dd format, then this field contains program-specific data whose format has been negotiated between Givex and the merchant. This should not normally be a surprise to the integration.

Effective Date or PSD Indicator:

For coupon orders which become effective at some date in the future, the effective date field provides that date, in yyyy-mm-dd format. If the field contains data which is not a date in that format, it is an indicator that the "Balance as of Effective Date" field contains program-specific data whose format has been negotiated between givex and the merchant. Program-specific data should only be returned in response to a specific request from the integration, but integrations should be prepared to ignore the Effective Date and Balance as of Effective date field if a PSD indicator is present.

Currency code:

A code that specifies the currency of the returned balance. This code is taken from the ISO currency code list. For example "CAD" can be used for Canadian funds and "USD" can be used for U.S. funds.

Promo code:

A code defined by the merchant that affects the calculation from amount and units to points.

Cashout ID:

A number assigned to a close request.

Cashout count:

The number of transactions affected by a close request.

Member name:

This text field will be populated with the loyalty member’s first, middle and last names separated by colons. If there is no member name associated with the certificate then an empty string is returned in this field.

Receipt message:

This field may be populated with a text message designed to be printed on the POS receipt. If the field contains text it should be printed, if empty the field should be ignored. The maximum length of any one line will be 40 characters. Lines will be terminated by newline (0x0A) characters, and multi-line messages should be printed in the correct format by the POS. Text should be printed centered on the receipt.

Total check amount:

This field must be populated with the total value of the check or order. Specify in the currency of the transaction location with decimal, eg. "123.00".

SKU:

This field consists of a sku number, price, and quantity. The format is as follows : SKU<US>Price<US>Quantity. The Quantity element is optional, if it is not included the value will be assumed as one. For subsequent instances of SKUs a <GS> character is used. Example :

19999<US>19.99<US>3<GS>20000<US>4.99<GS>19999<US>19.99

19999 - Price : 19.99, Qty : 4
20000 - Price : 4.99, Qty : 1

Verify Response

This value indicates if a customer’s card number has been registered with their email address and/or name in the Givex system. The value ranges from 0 to 4 (see below) and only one such value will be returned in response.

0 - Certificate is registered. If provided, email address and first/last
    name match.
1 - Certificate is registered. First/Last name does not match.
2 - Certificate is registered. Email address does not match.
3 - Certificate is registered. First/Last name and email address do not
    match.
4 - Certificate is not registered.

Email address:

This text field must be populated with the customer’s email address.

First Name:

This text field will be populated with the customer’s first name.

Last Name:

This text field will be populated with the customer’s last name.

Discount amount:

This field will be populated with the dollar discount amount for any applicable discounts.

Member title:

This text field will be populated with the loyalty member’s title and must have a value of "Mr", "Mrs", "Ms", "Miss", "Dr", "Rev", "Sr", "Mx" or "Other".

Member fname:

This text field will be populated with the loyalty member’s first name. If there is no first name associated with the loyalty number then this field will be blank.

Member mname:

This text field will be populated with the loyalty member’s middle name. If there is no middle name associated with the loyalty number then this field will be blank.

Member lname:

This text field will be populated with the loyalty member’s last name. If there is no last name associated with the loyalty number then this field will be blank.

Member address:

This text field will be populated with the loyalty member’s address, city, postal code, country, county and province (optional) +, separated by a pipe symbol, |. The country should be specified using the ISO 3166-1-alpha-2 code. For example, "23 Anyroad St.,Suite 100|London|GU1 3UY|UK|Greater London".

Member mobile:

This text field will be populated with the loyalty member’s mobile number. The format of the phone number must conform to the ITU standard and the National Numbering Plan.

Member email:

This text field will be populated with the loyalty member’s email address.

Member birthdate:

This field will be populated with the loyalty member’s date of birth. The date must be in the format YYYY-MM-DD, where YYYY is the four-digit representation of the year, MM is the two-digit month ranging from 01 to 12 and DD is the two-digit day ranging from 01 to 31.

SMS Contact answer:

This field is used to indicate the loyalty member’s permission to be included in SMS campaigns initiated by the merchant. A Y or y in this field means the member has granted permission; a N or n means otherwise.

Email contact answer:

This field is used to indicate the loyalty member’s permission to be contacted via email. A Y or y in this field means the member has granted permission; a N or n means otherwise.

Mail contact answer:

This field is used to indicate the loyalty member’s permission to be contacted via mail post. A Y or y in this field means the member has granted permission; a N or n means otherwise.

Referring member number:

This is the primary reference number for the referring member. This number will be the number encoded on the member’s loyalty card.

Member phone:

This text field will be populated with the loyalty member’s phone number. The format of the phone number must conform to the ITU standard and the National Numbering Plan.

Referring member name:

This will be populated with the referring member’s first and last name in that order. If there is no referring member or the referring members name is empty, this field will be blank.

Amenities:

The amenities field consists of amenity codes indicating which amenities a store has available. If multiple amenities are available, a <GS> character is used as a separator.

customerType

Recipient email is:

The email address of the e-cert recipient when delivered from Givex system. It should be the value from ecertOrderInfo ⇒ orderDetailInfo ⇒ receivingEmail. This is currently not used for e-certs using Bulk E-certificate Pickup Templates as these may be sent from third party email platforms.

Customer Login:

A CWS customer’s login ID. This will normally be provided by the consumer. A login token can be returned by 958 (CWS Verify Login), and used in place of the Customer Login for all calls except 947 (Update Account), 953 (Update Password), 954 (Reset Password) and 958. The token should be sent to Givex exactly as it was returned from that call. Note that the token will typically expire after 15 minutes of inactivity. This inactivity period can be configured, however for security reasons it is better not to use very long periods for this sort of expiry. The Login Token may become invalid before the end of the inactivity period. Integrators should be prepared to handle error codes 150 and 249 when using login tokens.

Customer Password:

A CWS customer’s password. This will normally be provided by the consumer. In case a login token is being used in place of the Customer Login, Customer Password should be left blank. The Givex system can be configured to allow certain calls to not require the Customer Password. An integration designed to work with this configuration should leave the password blank for such calls.

CardList:

Found in CWS Account Card List response message. The CardList element represents cards that are registered to the CWS account. The CWS Account Card List service returns zero or more instances of CardList elements.

Possible values for cardStatus are 0 (inactive), 1 (active), and 2 (on hold). Cards that are expired or have a cardStatus greater than 2 won’t be returned. Possible values for Auto Reload Status are t (there is an active reload schedule for this card) or f(there isn’t an active reload schedule for this card).

5.0 Transaction Message Formats

All fields in the services described below are required except those marked as optional. Optional fields should either be included with no data or omitted.

Most of the services return a result code which is zero for success and non-zero for failure. Upon failure an error message is also returned. Please note that this text message is meant for display only and may be changed without notice. If you need to programmatically inspect the error returned the result code should be used and not the error message itself.

NOTE - Applications should always be designed to handle future additions to the protocol services defined below. This means not sending extra fields to Givex and to ignore any unexpected fields added to the return packet.

General Service Rules:

  • When performing a transaction, the Givex number must have been issued to the merchant for which the user ID is assigned (unless the merchants have been grouped administratively).

Specifying a value that falls outside this range will result in an error (10 - Invalid amt < min or 11 - Invalid amt > max).

  • The password must be included as a blank field if there is no value for the password.

6.2 Appendix B - Description of Givex (generated) numbers

A Givex (generated) number is constructed as follows:

<ISO IIN> <validation code> <serial number> <check digit>

ISO IIN: This field will be the Givex or the merchant’s ISO IIN.

Validation code: A five digit random number.

Serial number: Sequential number referring to the certificate in the Givex database. This number should be displayed on receipts since there is not enough data in it to redeem the certificate. Integrators should allow for this number to be up to 10 digits long.

Check digit: The check digit is a single digit to allow terminals to check for single digit errors. It is an alternate weighted mod 10 checksum. It is calculated on the entire Givex number except for the check (last) digit itself alternating a weight of 1 and 2 starting with 1 on the rightmost digit and working backwards. Here is some C code to calculate it.

static void
mk_chksum(char *s)
{
    int     mult = (strlen(s) % 2) ? 0 : 1;
    int     chksum = 0;
    int     k;
while (*s)
{
    k = *s - '0';
    chksum += k;
if (mult)
{
    chksum += k;
    mult = 0;
}
else
    mult = 1;
    s++;
}
    *s++ = (chksum % 10) + '0';
    *s = 0;
}
